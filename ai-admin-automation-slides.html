My Role in This Project
我不是在寫程式
我在把問題變成可執行的規格

這個專案我扮演的是需求端與開發端之間的橋樑：釐清客戶真正的問題、把它量化成可衡量的目標、轉譯成 AI 團隊能執行的規格，並在交付過程中持續驗證方向是否正確。

01
專案背景

客戶是誰
問題是什麼

三間清潔公司，共同業主，員工約 10 人外勤作業。每月產生大量發票單據，全靠月底人工彙整，老闆無法即時掌握支出狀況。

3間公司共用流程
月底才能看到帳
$0可用預算
02
問題量化

把「感覺很麻煩」
變成可以衡量的數字

跟客戶訪談後，把模糊的「帳務很亂」轉化成具體的痛點指標：

問題量化
月底彙整時間每月 4–6 小時人工輸入
能見度落差支出狀況延遲 30 天才知道
跨公司查帳需切換 3 份不同 Excel
員工報帳紙本易遺失，無留存紀錄

量化的目的不是為了漂亮的數字，而是讓解法設計有明確的優先順序。

03
需求轉譯

從業主語言
翻譯成技術規格

業主說的是「希望員工拍一張照片就好」，我把它拆成 AI 團隊能執行的規格：

業主說的

  • 不要讓員工學新東西
  • 老闆隨時能看帳
  • 不要花錢
  • 三間公司要在一起看

轉譯後的規格

  • 報帳入口：手機掃 QR，零安裝
  • 看板：密碼保護，任意裝置可開
  • 架構：Vercel + Sheets，月費 $0
  • 多公司：同一系統切換篩選
04
向 AI 團隊溝通

我給開發團隊的
不是想法，是邊界

跟 AI 開發團隊協作時，我的角色是設定清楚的邊界條件，讓執行方向不偏移：

必須做什麼
→
不能做什麼
→
驗收標準是什麼

例如：「員工報帳入口不能要求登入帳號」「三間公司要在同一個看板切換，不是三個網址」「部署成本必須是零」——這些都是我在開發前就定義好的限制。

05
方向校正

我在執行過程中糾正了一個關鍵方向

AI 團隊初版把 Gemini AI 放在架構核心，作為主要的發票辨識引擎。我在驗收時發現這個判斷是錯的。

AI 團隊的初始設計

Gemini Vision API 辨識所有發票照片 → 萃取金額、日期、廠商

問題：台灣電子發票本來就有標準 QR Code，根本不需要 AI 看圖。AI 只在手寫收據才有作用，而手寫收據是少數情況。

我要求調整後的架構

電子發票走 QR Code 直接解析 → Gemini 只處理無 QR 的手寫收據

結果：主流程更穩定、不依賴 AI 準確率、延遲更低，且符合實際使用情境。

這次校正讓我確認了一件事：「用了 AI」和「AI 是對的解法」不是同一件事。 顧問的價值在於能在技術執行前就把這個問題問清楚。

06
風險識別

我主動提出的
三個安全風險

系統跑通後，我要求進行完整的 code review，並識別出三個開發團隊未主動揭露的高風險問題：

  • 寫入端無驗證：任何人知道 API URL 就能寫進客戶帳本
  • 預設密碼公開在 repo：環境變數漏設時系統直接用公開密碼
  • 財務資料可能被誤改：作廢功能用位置索引操作，試算表排序後會改錯列

這三個問題都在交付前修補完畢。顧問的責任不只是讓功能跑通，而是確保它跑通的方式是安全的。

07
驗收設計

我設計了驗收標準
而不是只看「能不能開」

要求 AI 團隊撰寫端對端測試腳本，用真實資料流驗證整套系統，而不是只靠「畫面能打開」來判定完成。

驗收項目結果
支出寫入 → 看板顯示✅ 3 筆，含作廢狀態
收入記錄結構✅ 2 筆，合計正確
應收帳款狀態切換✅ 待收 / 已收正確
所有欄位結構對齊✅ 24 / 24 assertions 通過

09
顧問反思

這個案子讓我改變的
一個工作習慣

安全風險是事後 code review 才發現的，不是我在需求階段就定義清楚的。

對於處理真實財務資料的系統，安全邊界應該是需求規格的一部分，而不是最後才補的 checklist。

下一次，我會在需求文件裡直接加一欄：「誰可以寫入、誰可以讀取、什麼操作需要驗證」——在開發開始前就確認清楚，而不是等系統跑通後才問。
